## Problem Statement

信息碎片化导致知识和机会流失。作为一个信息密集型工作者（创始人），每天的微信聊天、Get笔记、照片散落在不同工具里，无法被系统性利用。既浪费大量时间回翻聊天记录和笔记，又因为找不到关键信息而错过商业机会。

核心痛点不是"没地方存信息"——而是"信息存了但没法被利用"。

## Demand Evidence

- **第一个用户是创始人自己** — 每天被这个问题困扰，既浪费时间又错过机会
- 尚未有外部付费用户验证
- 目标用户画像：信息密集型工作者（创始人、投资人、BD、咨询顾问）
- 现有替代方案（微信收藏 + Get笔记 + 相册）是凑合的流程，工具之间没有连接

## Status Quo

当前"凑合流程"：
- 微信重要消息 → 收藏或手动复制
- 灵感想法 → Get笔记 (biji.com) 记录
- 照片 → 手机相册
- **关键问题：** 这三个渠道之间完全断开，没有交叉关联，没有自动蒸馏，没有主动提醒

真正的竞争对手不是 Notion 或 Obsidian——是"凑合着用 + 靠大脑记忆"。

## Target User & Narrowest Wedge

**Target User:** 创始人自己（dogfooding）以及类似画像的信息密集型专业人士

**Narrowest Wedge:** Get笔记 → AI 蒸馏
- 连接 biji.com OpenAPI
- AI 自动分类、标签、实体提取、摘要
- 知识持续积累，Agent 可查询历史
- 选择 Get笔记作为起点因为：有 API（零导入摩擦）、已有数据（立刻能看到效果）

## Constraints

- 完全本地存储（隐私敏感，不上云）
- Python 技术栈
- Claude API 做蒸馏（成本需关注）
- 文件系统存储（Markdown + JSON），不用数据库
- 一个人的项目，需要在 3-4 小时内完成 MVP

## Premises

1. 核心问题是"信息碎片化导致知识和机会流失"
2. 这个问题也困扰其他信息密集型工作者（创始人、投资人、BD、顾问）
3. 最有价值的切入点是"蒸馏"而不是"存储"——用户不缺存放信息的地方，缺的是把信息变成可行动洞察的能力
4. Get笔记 API 是最低摩擦的数据源，应该作为 MVP 的起点
5. 产品价值主张："你的 AI 参谋——它读了你写的所有东西，帮你发现你自己都忘了的线索"

## Approaches Considered

### Approach A: "每日智能摘要" (Minimal Viable)
- 连接 API，每天生成摘要推送
- Effort: S (CC: ~1小时)
- Pros: 零操作获得价值，最快验证
- Cons: 不积累知识，每天独立

### Approach B: "AI 参谋知识库" (Ideal Architecture) ← CHOSEN
- 完整 Phase 1：API 同步 + 蒸馏管道 + 实体提取 + 知识库存储
- Effort: M (CC: ~3-4小时)
- Pros: 知识持续积累，compounding value
- Cons: 验证周期稍长

### Approach C: "笔记对话" (Creative/Lateral)
- 对话式检索，不做蒸馏
- Effort: S (CC: ~1.5小时)
- Pros: 直接解决"找不到信息"
- Cons: 被动查询，不主动发现

## Recommended Approach

**Approach B: AI 参谋知识库**

理由：AI 编程把 M 级工作量压缩到 3-4 小时，而 B 是唯一能持续积累价值的方案。创业产品最重要的特性是 compounding value — 用户用得越久，切换成本越高，价值越大。A 和 C 能快速验证但不 compound。

实现计划（对标 CEO Review Phase 1-5）：
1. Get笔记 API 同步 → inbox
2. AI 蒸馏管道（分类/标签/实体提取/摘要）
3. 知识库存储（Markdown + YAML frontmatter）
4. Staging 机制（防幻觉）
5. CLI 查询接口
6. 后续：微信导入、照片导入、知识图谱、智能层

## Open Questions

1. **Get笔记 API 具体能力？** 需要研究 biji.com/openapi 的文档，了解能拉取什么数据、分页、认证方式
2. **蒸馏质量预期？** 第一版蒸馏出来的实体和关联准确率如何，需要实际数据验证
3. **商业模式？** 如果作为创业项目，付费模式是什么？SaaS 订阅？按 API 调用量？
4. **其他信息密集型用户的具体痛点是否与创始人相同？** 需要跟 2-3 个潜在用户聊聊

## Success Criteria

1. **自己每天在用** — dogfooding 是最重要的验证
2. **能从历史笔记中找到自己忘了的关联** — "哦对，3周前记过这个"
3. **减少每周花在信息整理上的时间** — 可衡量
4. **找到 2-3 个愿意试用的其他用户** — 验证是否不只是自己的需求

## The Assignment

**不要先写代码。**

本周做这一件事：**找 3 个跟你一样信息密集的人（创始人、投资人、或 BD），问他们一个问题："你上一次因为找不到之前记过的某条信息而错过了什么？"**

如果 3 个人中有 2 个能给你一个具体的、痛苦的例子——你就有了真正的需求验证。如果他们想了半天说"好像没有"——你可能在解决一个只有你自己才有的问题（这不一定是坏事，但你需要知道）。

带着这 3 个对话的结果回来，我们再决定是先给自己建 MVP，还是调整方向。

## What I noticed about how you think

- 你在 CEO Review 时对每一个扩展项都说了 "加入计划"——7/7 全部接受。这说明你有大愿景和不害怕 scope 的勇气。但创业最重要的是说 "不"。你需要练习砍掉自己喜欢的东西。
- 你选择 Get笔记 API 作为切入点而不是微信——这是对"最低摩擦"的正确判断。你本能地在找阻力最小的路径，这是好的创业直觉。
- 当我追问具体痛点例子时你选择了跳过——我理解你想往前推进，但"说不出具体例子"本身是一个信号。最好的创业想法来自你能用血和泪讲出来的故事。找到那个故事。
- 你把产品定义为"Agent-Native"而不是"给人看的 UI"——这个框架选择本身很有品味。你在思考下一代产品应该长什么样，而不是复制现有模式。
